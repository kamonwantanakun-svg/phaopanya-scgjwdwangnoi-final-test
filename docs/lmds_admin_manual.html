1. ภาพรวมระบบ LMDS

LMDS (Logistics Management Data System) V5.5 เป็นระบบจัดการข้อมูลลอจิสติกส์ที่ทำงานบน Google Apps Script และ Google Sheets โดยมีจุดประสงค์หลักคือการรับข้อมูลการจัดส่งจาก SCG API แล้วจับคู่ชื่อปลายทาง (ShipToName) กับพิกัด GPS ที่ถูกต้อง รวมถึงสร้าง Master Data ของบุคคล สถานที่ พิกัด และจุดหมายปลายทาง เพื่อให้สามารถติดตามและวิเคราะห์การจัดส่งได้อย่างมีประสิทธิภาพ

ระบบแบ่งออกเป็น 2 กลุ่มการทำงานหลัก ได้แก่ กลุ่ม 1: Master Data Pipeline ซึ่งรับผิดชอบการล้างข้อมูล การจับคู่ และการสร้าง Master Data และ กลุ่ม 2: Daily Operations ซึ่งรับผิดชอบการดึงข้อมูลจาก SCG API การจับคู่พิกัดสำหรับงานประจำวัน และการสร้างรายงานสรุป

สถาปัตยกรรมระบบ

ระบบ LMDS ประกอบด้วยไฟล์ Source Code ทั้งหมด 22 ไฟล์ แบ่งเป็น 3 กลุ่มตามโฟลเดอร์ ดังนี้:

  • 0_core_system/ — ไฟล์แกนกลาง 6 ไฟล์: 00_App.gs (Entry Point), 01_Config.gs (Configuration), 02_Schema.gs (Schema), 03_SetupSheets.gs (Sheet Setup), 14_Utils.gs (Utilities), 19_Hardening.gs (Audit & Security)
  • 1_group1_master_db/ — ไฟล์กลุ่ม 1 จำนวน 9 ไฟล์: รับผิดชอบการทำงานของ Pipeline ตั้งแต่โหลดข้อมูลดิบ จนถึงจับคู่และสร้าง Master Data
  • 2_group2_daily_ops/ — ไฟล์กลุ่ม 2 จำนวน 7 ไฟล์: รับผิดชอบการดึงข้อมูล SCG API จับคู่พิกัด และรายงานสรุปประจำวัน

3. กลุ่ม 1: Pipeline ล้างข้อมูล & Master Data

กลุ่ม 1 เป็น Pipeline หลักของระบบที่ทำหน้าที่รับข้อมูลดิบจากชีต Source (SCG) แล้วจับคู่กับ Master Data ที่มีอยู่ หรือสร้างรายการใหม่หากไม่พบข้อมูลที่ตรงกัน กระบวนการนี้ประกอบด้วย 3 ขั้นตอนหลักที่ต้องทำตามลำดับ คือ โหลดข้อมูลดิบ → Normalize → Match Engine

3.1 ขั้นตอนการรัน Pipeline

สามารถรัน Pipeline ได้ 2 วิธี คือรันทั้งหมดในครั้งเดียว หรือรันทีละขั้นตอน การรันทีละขั้นตอนมีประโยชน์เมื่อต้องการตรวจสอบผลระหว่างขั้นตอน หรือเมื่อต้องการรันเฉพาะบางขั้นตอนที่ล้มเหลว

Step 1: โหลดข้อมูลดิบ → Step 2: Normalize → Step 3: Match Engine → Review Queue

Step 1: โหลดข้อมูลดิบจากแหล่ง (runLoadSource)

ขั้นตอนนี้จะอ่านข้อมูลจากชีต Source (SCG) และกรองเฉพาะแถวที่ยังไม่ได้ประมวลผล (SYNC_STATUS ไม่ใช่ SUCCESS และไม่ใช่ REVIEW) ระบบจะตรวจสอบ Invoice No ที่มีอยู่ใน FACT_DELIVERY แล้วข้ามแถวที่เคยประมวลผลแล้วโดยอัตโนมัติ พร้อมอัปเดต SYNC_STATUS เป็น SUCCESS ให้

Step 2: Normalize ชื่อ/ที่อยู่ (runNormalize)

ขั้นตอนนี้จะทำความสะอาดข้อมูลชื่อบุคคลและที่อยู่ผ่านโมดูล 05_NormalizeService.gs ซึ่งประกอบด้วย 7 ขั้นตอนสำหรับชื่อบุคคล ได้แก่: แยกเบอร์โทร, แยกเลขเอกสาร, แยกหมายเหตุการจัดส่ง, ตรวจชื่อบริษัท, ตัดคำนำหน้า, ลบอักขระพิเศษ และสร้างผลลัพธ์ Normalized

Step 3: Match Engine (runMatchEngine)

ขั้นตอนนี้เป็นหัวใจของ Pipeline โดยจะนำข้อมูลที่ผ่านการ Normalize แล้วไปจับคู่กับ Master Data ที่มีอยู่ ระบบจะใช้ LockService เพื่อป้องกันการรันซ้ำซ้อน และมี Time Guard จำกัดเวลาทำงานไม่เกิน 5 นาที (300,000 ms) หากเวลาใกล้หมด ระบบจะ flush ข้อมูลที่ประมวลผลแล้วและติดตั้ง Auto-Resume Trigger เพื่อรันต่อในครั้งถัดไปโดยอัตโนมัติ

3.2 Match Engine: การตัดสินใจจับคู่

Match Engine ใช้ระบบ Scoring เพื่อตัดสินใจว่าข้อมูลแต่ละรายการควรจับคู่อย่างไร โดยพิจารณาจากคะแนนความคล้ายคลึง (Confidence Score) เทียบกับค่าขีดแบ่ง (Threshold) ที่กำหนดไว้ใน AI_CONFIG ดังนี้

คะแนนการตัดสินใจผลลัพธ์
≥ 90AUTO_MATCHจับคู่อัตโนมัติ เขียนลง FACT_DELIVERY ทันที
70–89NEEDS_REVIEWส่งเข้าคิว Q_REVIEW รอแอดมินวินิจฉัย
50–69IGNOREไม่พิจารณา (คะแนนต่ำเกินไป)
< 50CREATE_NEWสร้างรายการใหม่ใน Master Data

เมื่อ Match Engine ทำงานเสร็จ จะมีการเขียน Alias ใหม่ลงใน M_ALIAS โดยอัตโนมัติผ่าน autoEnrichAliasesFromFactBatch_() ซึ่งเป็น Single Writer สำหรับ M_ALIAS ใน Auto Pipeline (ห้ามเรียก createGlobalAlias() ใน Pipeline อัตโนมัติ)

3.3 Review Queue: คิวรอตรวจสอบ

เมื่อ Match Engine พบรายการที่มีคะแนน 70–89 จะถูกส่งเข้าชีต Q_REVIEW ซึ่งเป็นคิวรอให้แอดมินวินิจฉัย ระบบมี Dropdown ให้เลือก Decision ได้ 4 ตัวเลือก ดังนี้

Decisionคำอธิบายผลลัพธ์
CREATE_NEWสร้างรายการใหม่ใน Master Dataสร้าง Person/Place/Geo/Dest ใหม่ + เขียน FACT_DELIVERY
MERGE_TO_CANDIDATEผสานเข้ารายการที่มีอยู่ใช้ Person/Place/Geo ที่ตรงกัน + เขียน FACT_DELIVERY
ESCALATEส่งต่อให้ผู้มีอำนาจสูงกว่าเปลี่ยน Priority เป็นระดับสูง
IGNOREไม่สนใจรายการนี้ข้ามรายการ ไม่บันทึกลง FACT_DELIVERY

เมื่อเลือก Decision ในชีต Q_REVIEW ระบบจะผ่าน onEdit() trigger ประมวลผลทันที และทาสีแถวตามผลลัพธ์ นอกจากนี้ยังมีเมนู "รันคำสั่งที่เลือกไว้ทั้งหมด" (applyAllPendingDecisions) สำหรับดำเนินการทีเดียวหลายรายการ โดยมี Time Guard ป้องกัน Timeout เช่นเดียวกับ Match Engine

หลังจากเลือก Decision ใน Q_REVIEW ระบบจะประมวลผลทันทีผ่าน onEdit Trigger ไม่จำเป็นต้องรันเมนูเพิ่มเติม ยกเว้นกรณีต้องการประมวลผลหลายรายการพร้อมกัน

4. กลุ่ม 2: งานประจำวัน (SCG)

กลุ่ม 2 รับผิดชอบการดึงข้อมูลการจัดส่งจาก SCG API และจับคู่พิกัด GPS สำหรับงานประจำวัน โดยมี 18_ServiceSCG.gs เป็น Commander ที่ควบคุมขั้นตอนทั้งหมด ตั้งแต่ดึงข้อมูล จับคู่พิกัด ไปจนถึงสร้างรายงานสรุป

4.1 ดึงข้อมูล SCG API (fetchDataFromSCGJWD)

ขั้นตอนนี้จะดึงข้อมูลการจัดส่งจาก SCG JWD API ผ่าน URL https://fsm.scgjwd.com/Monitor/SearchDelivery โดยใช้ Cookie ที่เก็บใน PropertiesService หรือชีต Input (เซลล์ B1) ข้อมูลที่ได้จะถูกเขียนลงชีต "ตารางงานประจำวัน" พร้อมตรวจสอบ GPS Quality และทาสีแถวตามผลการตรวจสอบ

ก่อนใช้งาน ต้องตั้งค่า Cookie ผ่านเมนู กลุ่ม 2 → ตั้งค่า SCG Cookie หรือ ระบบ → ตั้งค่า SCG Cookie เนื่องจาก Cookie มีอายุจำกัด หากดึงข้อมูลไม่สำเร็จ อาจต้องตั้งค่า Cookie ใหม่

4.2 จับคู่พิกัด — 2-Tier Search

ระบบค้นหาพิกัด GPS จาก ShipToName ใช้นโยบาย ShipToName-Only (ตั้งแต่ v5.4.003) ซึ่งใช้เฉพาะชื่อปลายทาง (ShipToName) เป็นฐานในการค้นหาเท่านั้น โดยไม่ใช้ ShipToAddress หรือ LatLong_SCG เนื่องจากข้อมูลเหล่านั้นไม่น่าเชื่อถือ

สถาปัตยกรรม 2-Tier Search

Tierชื่อกลไกความเร็ว
Tier 0 M_ALIAS Fast Track ShipToName → normalize → M_ALIAS reverse index → masterUuid → Dest → lat,lng เร็วที่สุด (O(1) hash lookup)
Tier 1 Person Anchor ShipToName → resolvePerson() → getDestsByPersonId() → usage-dominant Dest ปานกลาง (DB lookup)
NOT_FOUND ไม่พบข้อมูล เว้นว่าง LatLong_Actual — ไม่มี fallback ใดๆ —

เมื่อรัน applyMasterCoordinatesToDailyJob ระบบจะวนทุกแถวใน "ตารางงานประจำวัน" และเรียก findBestGeoByPersonPlace() สำหรับแต่ละ ShipToName ผลลัพธ์จะถูกเขียนลงคอลัมน์ LatLong_Actual พร้อมทาสีแถว:

  • เขียว พบพิกัด (FOUND / FOUND_DOMINANT / FOUND_ALIAS_FAST)
  • แดง ไม่พบพิกัด (NOT_FOUND)
หากรันแล้วเกิด Timeout (เกิน 5 นาที) ระบบจะติดตั้ง Auto-Resume Trigger ให้รันต่อโดยอัตโนมัติ แต่หากต้องการรันเอง ให้รันเมนู "จับคู่พิกัด" อีกครั้ง — ระบบจะข้ามแถวที่มีพิกัดดีอยู่แล้ว

4.3 สรุปเจ้าของสินค้า & Shipment

หลังจากจับคู่พิกัดเสร็จ ระบบจะสร้างรายงานสรุป 2 ชีตให้อัตโนมัติ:

  • สรุป_เจ้าของสินค้า — สรุปจำนวน Invoice แยกตาม SoldToName (เจ้าของสินค้า) พร้อมจำนวน E-POD ที่ตรวจพบ โดยเช็คจากรายชื่อ EPOD_OWNERS: BETTERBE, SCG EXPRESS, เบทเตอร์แลนด์, JWD TRANSPORT
  • สรุป_Shipment — สรุปจำนวน Invoice แยกตาม ShipmentNo และทะเบียนรถ พร้อมจำนวน E-POD

5. ระบบ & ตั้งค่า

5.1 การตั้งค่า API Key & Cookie

Gemini API Key (setupEnvironment)

ระบบใช้ Gemini API (รุ่น gemini-1.5-flash) สำหรับการวิเคราะห์ข้อมูล แม้ว่าปัจจุบัน AI Reasoning ถูกปิดใช้งาน (AI_CONFIG.USE_AI_REASONING = false) แต่ยังคงต้องตั้งค่า API Key เพื่อให้ระบบผ่านการตรวจสอบ Preflight Audit คีย์จะถูกเก็บใน Script Properties ผ่าน PropertiesService.getScriptProperties() โดยตรวจสอบรูปแบบว่าขึ้นต้นด้วย AIza และมีความยาว 35 ตัวอักษร

SCG Cookie (setSCGCookie_UI)

Cookie สำหรับเชื่อมต่อ SCG API จะถูกเก็บใน Script Properties ผ่านคีย์ SCG_COOKIE (แนะนำ) หรือในเซลล์ B1 ของชีต Input (วิธีเก่า) หากใช้วิธีเก่า ระบบจะอ่านจากเซลล์ B1 แล้วย้ายไปเก็บใน PropertiesService ให้อัตโนมัติ Cookie มีอายุจำกัด หากดึงข้อมูล SCG ไม่สำเร็จ ให้ตั้งค่า Cookie ใหม่

5.2 การจัดการรายชื่อ Admin (setupAdminList_UI)

ระบบมี Authorization Guard ที่ตรวจสอบสิทธิ์ผู้ใช้ผ่านฟังก์ชัน isAuthorizedUser_() ก่อนให้รันฟังก์ชันที่สำคัญ เช่น setupAllSheets, generatePersonAliasesFromHistory, applySheetProtection_UI รายชื่อ Admin จะถูกเก็บใน Script Properties ผ่านคีย์ ADMIN_EMAILS เป็น comma-separated string

5.3 Alias System & Migration

ระบบ LMDS V5.5 ใช้ Hybrid Alias Architecture ที่มี M_ALIAS เป็นตารางกลางสำหรับเชื่อมโยงชื่อสกปรก/ย่อ/ผิดไปยัง master_uuid ซึ่งสามารถแปลงเป็น Entity ID (personId/placeId) แล้วค้นหาพิกัดได้ เป็น Single Source of Truth สำหรับ Alias Resolution ที่ Group 2 ใช้ค้นหา

Auto Pipeline เขียน M_ALIAS ที่ autoEnrichAliasesFromFactBatch_() เท่านั้น ห้ามเรียก createGlobalAlias() ใน Pipeline อัตโนมัติ — ฟังก์ชันนี้ใช้สำหรับ Migration/Admin เท่านั้น

Migration: Hybrid Alias System (MIGRATION_HybridAliasSystem)

การ Migration แบ่งเป็น 5 ขั้นตอน ดังนี้:

  1. ตรวจสอบและเพิ่ม master_uuid ให้ M_PERSON และ M_PLACE
  2. ย้ายข้อมูลจาก M_PERSON_ALIAS → M_ALIAS
  3. ย้ายข้อมูลจาก M_PLACE_ALIAS → M_ALIAS
  4. เพิ่ม Canonical Name ของ Person และ Place เข้า M_ALIAS
  5. ดึงชื่อจาก FACT_DELIVERY → M_ALIAS

ระบบมี Checkpoint Resume ผ่าน PropertiesService ทำให้หาก Migration หยุดกลางคัน สามารถรันต่อจากขั้นตอนที่ค้างไว้ได้

5.4 การป้องกันข้อมูล Sensitive (applySheetProtection_UI)

ฟังก์ชันนี้จะตั้งค่า Protected Ranges และ Hidden Sheets สำหรับข้อมูลที่มี PII (Personally Identifiable Information) ดังนี้:

ชีตการป้องกันเหตุผล
ข้อมูลพนักงานProtected + Hiddenเลขบัตรประชาชน, เบอร์โทร
SCGนครหลวงJWDภูมิภาคProtected + Hiddenที่อยู่, Email, ชื่อลูกค้า
M_PERSONProtected (ไม่ซ่อน)ข้อมูลบุคคล (เบอร์โทร)
M_GEO_POINTProtected (ไม่ซ่อน)ข้อมูลพิกัด — เฉพาะ Script เท่านั้นที่เขียน

เฉพาะ Script Owner / Admin เท่านั้นที่สามารถแก้ไขชีตเหล่านี้ได้

7. การตรวจสอบระบบ (Audit & Hardening)

ระบบ LMDS มีเครื่องมือตรวจสอบหลายตัวสำหรับแอดมินใช้ตรวจสอบความพร้อมและคุณภาพของข้อมูล

Preflight Audit (runPreflightAudit)

ตรวจสอบความพร้อมของระบบก่อนรัน Pipeline โดยตรวจสอบ:

  • ชีตทั้งหมดมีอยู่ครบไหม (ตรวจจากรายการ SHEET ทั้งหมด)
  • จำนวนคอลัมน์ในแต่ละชีตตรงกับ Schema ไหม
  • GEMINI_API_KEY ตั้งค่าแล้วหรือยัง
  • มีแถวที่ไม่มี SYNC_STATUS ใน Source กี่แถว

ผลลัพธ์จะแสดงเป็นรายการ หากทุกอย่างปกติจะแสดง "ระบบพร้อมทำงาน 100%"

Detect Duplicates (detectDoubleProcessing)

ตรวจจับ Invoice ซ้ำใน FACT_DELIVERY โดย Normalize Invoice No แล้วนับจำนวนที่ปรากฏ หากพบ Invoice ซ้ำ จะแสดงรายการพร้อมจำนวนครั้งที่ซ้ำ (แสดงสูงสุด 10 รายการ)

System Integrity (checkSystemIntegrity)

ตรวจสอบว่าชีตที่จำเป็นทั้งหมดมีอยู่ใน Spreadsheet หรือไม่ รวมถึงชีต M_ALIAS และ M_PLACE_ALIAS ที่เพิ่มเข้ามาใน v5.4

Diagnostic (diagnoseSystemState)

วินิจฉัยสถานะ Pipeline ปัจจุบัน โดยแสดงข้อมูลสถิติจากหลายแหล่ง เช่น จำนวนแถวที่ประมวลผลแล้ว จำนวนที่รอ Review จำนวนที่ยังไม่ได้ประมวลผล

Fix Missing Sync Status (fixMissingSyncStatus)

ซ่อมแซมแถวในชีต Source ที่ไม่มี SYNC_STATUS โดยเติมค่า 'PENDING' ให้อัตโนมัติ ฟังก์ชันนี้ถูกเรียกผ่านระบบ ไม่มีเมนูโดยตรง แต่สามารถเรียกจาก Script Editor ได้

9. การจัดการ Cache

ระบบ LMDS ใช้ Cache 3 ชั้นเพื่อเพิ่มความเร็วในการทำงาน โดยแต่ละชั้นมีคุณสมบัติแตกต่างกัน

ชั้นที่ 1: RAM Cache (ใน Execution)

เป็น Cache ที่เก็บในหน่วยความจำระหว่างการทำงานของ Script Execution เดียวกัน มีความเร็วสูงสุด แต่จะหายไปเมื่อ Execution จบ ตัวแปรหลักได้แก่:

  • _SOURCE_ROWS_RAM_CACHE — แคชข้อมูล Source rows
  • _MAPS_SHEET_CACHE — แคชข้อมูล MAPS_CACHE sheet
  • _GLOBAL_GEO_DICT_CACHE — แคชฐานข้อมูลภูมิศาสตร์
  • _GLOBAL_GEO_DICT_CACHE_PLACE — แคชข้อมูล Place ภูมิศาสตร์
  • _GLOBAL_GEO_POINTS_CACHE — แคช Geo Points
  • _FACT_INVOICE_RAM_CACHE — แคช Invoice lookup
  • _GEO_LATLNG_RAM_CACHE — แคชพิกัด lat/lng
  • _ALIAS_ENRICHMENT_CONTEXT — แคช Alias enrichment

ชั้นที่ 2: CacheService (Google Cache)

เป็น Cache ที่เก็บใน Google CacheService มีอายุสูงสุด 6 ชั่วโมง (21,600 วินาที) และมีขนาดจำกัด 100 KB ต่อคีย์ ระบบใช้ Chunked Cache Pattern เพื่อรองรับข้อมูลที่เกิน 100 KB โดยแบ่งเป็นก้อนละ 200 รายการ

ชั้นที่ 3: Sheet Cache (Google Sheets)

เป็น Cache ที่เก็บในชีต MAPS_CACHE สำหรับผลลัพธ์ Geocoding เป็น Persistent Storage ที่ไม่หายไป เมื่อ RAM Cache และ CacheService ไม่มีข้อมูล ระบบจะอ่านจากชีตแทน

การล้าง Cache

เมนู "ล้างความจำระบบ (Clear Cache)" (invalidateAllGlobalCaches) จะล้าง RAM Cache ทั้งหมด รวมถึงเรียก invalidatePersonCache_(), invalidatePlaceCache_(), invalidateGeoCache_() จากโมดูลอื่นๆ ด้วย

ควรล้าง Cache เมื่อ: (1) แก้ไขข้อมูลในชีตโดยตรง, (2) รัน Pipeline ซ้ำหลังจากแก้ไข Master Data, (3) พบว่าข้อมูลที่แสดงไม่ตรงกับข้อมูลจริงในชีต

10. การแก้ปัญหาเบื้องต้น

Pipeline รันไม่ผ่าน / Timeout

หาก Pipeline รันไม่ผ่าน อาจเกิดจาก Time Limit (5 นาที) ระบบจะติดตั้ง Auto-Resume Trigger ให้อัตโนมัติ แต่หากต้องการรันเอง:

  1. รันเมนู "ล้างความจำระบบ" เพื่อล้าง Cache เก่า
  2. รันเมนู "Step 3 — Match Engine" อีกครั้ง — ระบบจะข้ามแถวที่ประมวลผลแล้วอัตโนมัติ
  3. หากยังไม่ผ่าน ให้ตรวจสอบ SYS_LOG เพื่อดู Error message

จับคู่พิกัดไม่เจอ (สีแดงเยอะ)

หากจับคู่พิกัดในตารางงานประจำวันได้ผล NOT_FOUND เยอะ:

  1. ตรวจสอบว่า Alias ถูกสร้างครบไหม — รัน "สร้าง Alias อัตโนมัติจากประวัติ"
  2. ตรวจสอบ M_PERSON ว่ามีชื่อปลายทางอยู่ครบไหม
  3. รัน "Migration: Hybrid Alias System" หากยังไม่เคยรัน
  4. รัน "ดึงชื่อจาก SCG ดิบ → M_ALIAS" เพื่อเพิ่ม Alias จากข้อมูล SCG

ดึงข้อมูล SCG API ไม่สำเร็จ

หากดึงข้อมูล SCG ไม่สำเร็จ อาจเกิดจาก Cookie หมดอายุ:

  1. เปิดเบราว์เซอร์และล็อกอิน SCG JWD Portal
  2. คัดลอก Cookie จาก DevTools (Network → Request Headers → Cookie)
  3. รันเมนู "ตั้งค่า SCG Cookie" แล้ววาง Cookie ใหม่
  4. ลองดึงข้อมูลอีกครั้ง

ข้อมูลซ้ำใน FACT_DELIVERY

หากสงสัยว่ามีข้อมูลซ้ำ:

  1. รันเมนู "Detect Duplicates" เพื่อตรวจสอบ
  2. หากพบ Invoice ซ้ำ ให้ตรวจสอบแถวที่ซ้ำและลบรายการที่ไม่ถูกต้อง
  3. รัน "รีเซ็ตสถานะ SYNC" เฉพาะเมื่อต้องการให้ระบบประมวลผลใหม่ทั้งหมด (ใช้ด้วยความระมัดระวัง)

ชีตหาย / Header ผิด

หากชีตหายไปหรือ Header ไม่ตรงกับ Schema:

  1. รันเมนู "สร้างชีตทั้งหมด" — ระบบจะสร้างชีตที่หายไปและ Auto-Repair Header ที่ขาดหาย
  2. คอลัมน์ที่ถูกเพิ่มโดย Auto-Repair จะมีสีแดงอ่อนเพื่อให้สังเกตได้ง่าย

Smart Navigation ไม่ทำงาน
