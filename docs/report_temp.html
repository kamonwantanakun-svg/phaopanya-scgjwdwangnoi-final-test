1. ภาพรวมระบบ

LMDS คืออะไร

LMDS (Logistics Master Data System) คือระบบจัดการข้อมูลหลัก (Master Data) สำหรับงานขนส่งและโลจิสติกส์ ออกแบบมาเพื่อเป็น Single Source of Truth ของข้อมูลผู้ส่ง สถานที่ และการจัดส่ง ภายในระบบนิคมอุตสาหกรรมของ SCG Logistics ระบบนี้ทำหน้าที่รับข้อมูลดิบจาก SCG API จากนั้นทำความสะอาดข้อมูล (Data Cleansing) จับคู่กับ Master Record ที่มีอยู่ (Entity Resolution) และบันทึกผลลัพธ์ลงใน Google Sheets ซึ่งทำหน้าที่เป็นทั้ง Database และ User Interface ไปพร้อมกัน

กระบวนการหลักของ LMDS สามารถสรุปเป็น 3 ขั้นตอน: รับข้อมูลดิบจาก SCG API ผ่าน HTTP Request ที่เรียกใน Google Apps Script → ทำความสะอาดข้อมูลโดย Normalization Engine ที่จัดการปัญหาภาษาไทย เช่น คำนำหน้าชื่อ ที่อยู่ที่เขียนหลายรูปแบบ เบอร์โทรศัพท์ที่มีรูปแบบไม่มาตรฐาน → จับคู่กับ Master Record ด้วย Multi-strategy Match Engine ที่ใช้ 8 กฎในการตัดสินใจ ก่อน บันทึกผลลง FACT_DELIVERY และส่งรายการที่ไม่มั่นใจเข้า Q_REVIEW เพื่อให้มนุษย์ตรวจสอบ

2. สถาปัตยกรรม Trinity Framework

แกนหลักของ LMDS V5.5 คือสถาปัตยกรรม Trinity Framework ซึ่งแบ่งข้อมูล Master ออกเป็น 3 เสาหลัก (Pillars) แต่ละเสาทำหน้าที่ระบุตัวตนของมิติหนึ่งของข้อมูลขนส่งอย่างเป็นอิสระ ก่อนจะนำมารวมกันที่จุดตัด (Intersection) เพื่อสร้าง Destination Node ที่สมบูรณ์

Pillar ตาราง Master คอลัมน์ หน้าที่
WHO M_PERSON 10 ระบุตัวตนบุคคล — ชื่อ-นามสกุล คำนำหน้า เบอร์โทรศัพท์ อีเมล บริษัท รหัสพนักงาน ฯลฯ เสานี้ตอบคำถาม "ใครคือผู้เกี่ยวข้อง"
WHERE-Address M_PLACE 14 ระบุสถานที่ตามที่อยู่ — บ้านเลขที่ ซอย ถนน ตำบล อำเภอ จังหวัด รหัสไปรษณีย์ ชื่อสถานที่ ประเภทสถานที่ ฯลฯ เสานี้ตอบคำถาม "ที่ไหนตามที่อยู่"
WHERE-Coordinate M_GEO_POINT 14 ระบุพิกัด GPS — Latitude, Longitude, รัศมีความแม่นยำ (accuracy_radius_km) จังหวัดที่พิกัดอ้างอิง แหล่งที่มาพิกัด ระดับความมั่นใจ ฯลฯ เสานี้ตอบคำถาม "ที่ไหนบนแผนที่"
Intersection M_DESTINATION 11 จุดตัดของ 3 เสา — person_id + place_id + geo_id = 1 Destination Node ตารางนี้เชื่อมโยงบุคคล สถานที่ และพิกัดเข้าด้วยกัน พร้อมระบุประเภทจุดหมาย (Origin/Destination) และสถานะการใช้งาน

ทำไมต้องมี 3 เสา?

ข้อมูลขนส่งในโลกจริงมีความไม่สมบูรณ์ (Incomplete) อย่างเป็นระบบ ตัวอย่างเช่น:

เคส 1: ใบส่งของระบุชื่อผู้รับ "คุณสมชาย" แต่ไม่มีเบอร์โทร — เราสามารถระบุ Person ได้หากมีที่อยู่ที่ไม่ซ้ำ แต่ต้องแยก WHO ออกจาก WHERE จึงจะจับคู่ได้ถูกต้อง

เคส 2: ที่อยู่ "123 ถ.รัชดา" ปรากฏในหลายจังหวัด — หากรวม Address กับ Coordinate จะไม่สามารถแยกแยะได้ว่าเป็นสถานที่เดียวกันหรือไม่ จำเป็นต้องมี WHERE-Address และ WHERE-Coordinate แยกกัน

เคส 3: พิกัด GPS จากแอปส่งสินค้าอาจคลาดเคลื่อน 200 เมตร — ต้องใช้ Address เป็นหลักยึด (Anchor) ในการตรวจสอบ และใช้พิกัดเป็นตัวเสริม

การแยก 3 เสาช่วยให้ระบบสามารถจับคู่ข้อมูลได้ในสถานการณ์ที่ข้อมูลขาดหาย โดยแต่ละเสาสามารถ Match ได้อย่างอิสระ และนำผลมารวมกันที่ M_DESTINATION ซึ่งเป็นจุดตัดเดียวที่สร้างความสมบูรณ์ของข้อมูล นอกจากนี้ยังช่วยลดปัญหา Data Sparsity เพราะไม่จำเป็นต้องมีข้อมูลครบทุกมิติจึงจะสร้าง Destination ได้ — แม้จะมีเพียง Person + Place โดยไม่มี Coordinate ก็ยังสามารถสร้าง Destination Node ได้ในสถานะ Pending

3. Hybrid Alias Architecture

ปัญหาหลักของข้อมูลขนส่งคือ ชื่อเรียกที่หลากหลาย (Alias Variants) บุคคลคนเดียวกันอาจถูกเรียกหลายชื่อ เช่น "สมชาย ใจดี", "คุณสมชาย", "สมชาย ใจดี (SCG)" สถานที่แห่งเดียวกันอาจมีชื่อหลายรูปแบบ เช่น "โรงงาน SCG เมืองทอง", "SCG สำนักงานใหญ่", "The Siam Cement Group" LMDS V5.5 แก้ปัญหานี้ด้วย Hybrid Alias Architecture ที่ผสมผสาน Global Alias Ledger เข้ากับ Entity-specific Alias

Global Alias Ledger: M_ALIAS

ตาราง คอลัมน์ หน้าที่
M_ALIAS 8 ตารางกลางจัดการ Alias ข้ามโดเมน — เก็บ variant_name, master_uuid, domain (person/place), entity_id, source, confidence, is_active, created_at ช่วยให้ค้นหาชื่อย่อ/ชื่อเล่น/ชื่ออื่น แล้วเชื่อมโยงกลับไปยัง Master Record ได้ทันที
M_PERSON_ALIAS 6 Entity-specific Alias สำหรับบุคคล — เก็บ person_id, alias_name, alias_type (ชื่อเล่น/ชื่อย่อ/ชื่อเดิม), source, confidence, is_active ใช้สำหรับการค้นหาเฉพาะทางในโดเมน Person
M_PLACE_ALIAS 6 Entity-specific Alias สำหรับสถานที่ — เก็บ place_id, alias_name, alias_type (ชื่อเดิม/ชื่อย่อ/ชื่อทางการ), source, confidence, is_active ใช้สำหรับการค้นหาเฉพาะทางในโดเมน Place

master_uuid — Cross-domain Identity

แต่ละ Master Record (Person, Place, Geo) จะได้รับ master_uuid แบบ UUID v4 เมื่อสร้างครั้งแรก UUID นี้เป็นตัวระบุข้ามโดเมน (Cross-domain Identity) ที่ช่วยให้ M_ALIAS สามารถเชื่อมโยง Variant Name กลับไปยัง Entity ได้โดยไม่สนใจว่าจะเป็น Person หรือ Place กลไกนี้ช่วยลดความซ้ำซ้อนของตาราง Alias และทำให้สามารถค้นหาข้ามโดเมนได้ใน Query เดียว

Runtime Fast-path

ขั้นตอนการค้นหาแบบ Fast-path:

1. รับ Variant Name จากข้อมูลดิบ → 2. ค้นหาใน M_ALIAS → 3. ได้ master_uuid → 4. ใช้ master_uuid ค้นหา person_id หรือ place_id → 5. ส่งผลกลับทันทีโดยไม่ต้องผ่าน Match Engine เต็มรูปแบบ

Fast-path ช่วยลดเวลาประมวลผลจาก ~200ms (Full Match) เหลือ ~5ms (Alias Lookup) สำหรับชื่อที่เคยพบมาแล้ว

Single Writer Pattern

เพื่อป้องกันปัญหา Race Condition และ Data Inconsistency ระบบใช้ Single Writer Pattern โดยฟังก์ชัน autoEnrichAliasesFromFactBatch_() เป็นจุดเขียน M_ALIAS จุดเดียวในระบบ ฟังก์ชันอื่นที่ต้องการเพิ่ม Alias ใหม่จะต้องส่งผ่านฟังก์ชันนี้เท่านั้น ซึ่งจะทำการตรวจสอบ Duplicate, อัปเดต Confidence Score, และบันทึก Log ก่อนจะเขียนลงชีตจริง แนวทางนี้ช่วยให้มั่นใจได้ว่าข้อมูล Alias มีความสอดคล้องเสมอ แม้จะมีหลาย Process ทำงานพร้อมกัน

5. Pipeline Flow — 6 ชั้น

การประมวลผลข้อมูลใน LMDS V5.5 ดำเนินการผ่าน Pipeline 6 ชั้น (Six-Layer Pipeline) ซึ่งออกแบบตามหลัก Separation of Concerns แต่ละชั้นมีหน้าที่เฉพาะ และส่งผลลัพธ์ไปยังชั้นถัดไปอย่างเป็นระบบ การแบ่งชั้นแบบนี้ช่วยให้สามารถทดสอบแต่ละชั้นแยกกันได้ แก้ไขหรือปรับปรุงชั้นใดชั้นหนึ่งโดยไม่กระทบชั้นอื่น และติดตามตำแหน่งของปัญหาได้ง่ายเมื่อเกิดข้อผิดพลาด

Layer A

Ingestion — รับข้อมูลดิบ

อ่านข้อมูลดิบจาก SCG API ผ่าน HTTP Request กรอง Record ที่ไม่ถูกต้อง (เช่น ข้อมูลว่าง, รูปแบบไม่ตรง) แปลง JSON Response เป็น Internal Data Structure และตรวจสอบ Schema Conformance ชั้นนี้เป็นประตูทางเข้าเพียงจุดเดียวของระบบ ทำหน้าที่เป็น Gatekeeper ที่ป้องกันข้อมูลเสียไม่ให้เข้าสู่ Pipeline

Layer B

Normalization — ทำความสะอาดข้อมูล

ทำความสะอาดข้อมูลภาษาไทยโดยเฉพาะ: ลบคำนำหน้าชื่อ (คุณ, นาย, นาง, นางสาว) มาตรฐานที่อยู่ (จ. → จังหวัด, อ. → อำเภอ, ต. → ตำบล) จัดรูปแบบเบอร์โทร (0-XXXX-XXXX) และที่อยู่ให้เป็นรูปแบบเดียวกัน ชั้นนี้จัดการปัญหา Thai Language Edge Cases ที่ไม่พบในระบบภาษาอังกฤษ เช่น การเขียนชื่อตำบลหลายแบบ

Layer C

Master Resolution — จับคู่ Master Record

เป็นชั้นที่ซับซ้อนที่สุด ประกอบด้วย 3 ขั้นตอน: Candidate Search — ค้นหา Master Record ที่เป็นไปได้จาก M_PERSON, M_PLACE, M_GEO_POINT โดยใช้หลายกลยุทธ์ (Exact, Prefix, Fuzzy, Geo-radius) → Scoring — คำนวณคะแนนความเชื่อมั่นของแต่ละ Candidate โดยพิจารณาจาก Name Similarity, Address Match, Phone Match, Geo Proximity → Decision — ใช้ 8 กฎ (Match Rules) ในการตัดสินใจว่าจะ Accept, Reject หรือส่งเข้า Q_REVIEW

Layer D

Hybrid Alias — Fast Track Lookup

ก่อนจะเข้า Match Engine เต็มรูปแบบ ระบบจะตรวจสอบใน M_ALIAS ก่อนว่า Variant Name นี้เคยพบมาแล้วหรือไม่ หากพบจะใช้ Fast-path: variant name → M_ALIAS → master_uuid → person_id/place_id ทันที โดยไม่ต้องผ่าน Candidate Search และ Scoring ชั้นนี้ช่วยลดภาระของ Match Engine ได้มากกว่า 60% ใน Production Environment

Layer E

Transaction & Review — บันทึกผล

บันทึกผลลัพธ์ลง FACT_DELIVERY ด้วย Upsert Pattern (Insert หากยังไม่มี, Update หากมีอยู่แล้ว) สำหรับ Record ที่ Match Engine ไม่มั่นใจ (Confidence < Threshold) จะส่งเข้า Q_REVIEW เพื่อให้ Human Reviewer ตรวจสอบ ชั้นนี้เป็นจุดที่ระบบ "ยอมรับ" ว่า Automation ไม่สามารถตัดสินใจได้ทุกกรณี จึงต้องมี Human-in-the-loop เป็น Safety Net

Layer F

Governance & Hardening — กำกับดูแล

6. Match Engine — 8 Rules

Match Engine ของ LMDS V5.5 ใช้ 8 กฎการตัดสินใจ ที่จัดเรียงตามลำดับความสำคัญ (Priority) กฎที่มี Priority ต่ำกว่าจะถูกตรวจสอบก่อน หากเงื่อนไขตรงจะทำ Action ที่ระบุทันทีโดยไม่ต้องตรวจสอบกฎถัดไป การออกแบบแบบ Priority Chain นี้ช่วยให้มั่นใจว่ากฎที่เข้มงวดกว่าจะมีผลก่อนเสมอ

ลำดับ กฎ (Rule) Action เงื่อนไข Priority
1 INVALID_LATLNG REJECT พิกัด Latitude/Longitude อยู่นอกขอบเขตประเทศไทย (Lat < 5° หรือ > 21°, Lng < 97° หรือ > 106°) P1
2 LOW_QUALITY REVIEW ข้อมูลมีคุณภาพต่ำเกินไป (ไม่มีชื่อ, ไม่มีที่อยู่, ไม่มีเบอร์โทร) จนไม่สามารถ Match ได้อย่างมีความหมาย P2
3 GEO_PROVINCE_CONFLICT REJECT จังหวัดจากที่อยู่ไม่ตรงกับจังหวัดจากพิกัด GPS (ขัดแย้งกันอย่างชัดเจน) P3
4 NEARBY_PENDING REVIEW พบ Candidate ในรัศมีใกล้เคียง แต่สถานะเป็น Pending (ยังไม่ได้รับการยืนยัน) จึงต้องส่งให้ Reviewer ตรวจสอบ P4
5 FULL_MATCH ACCEPT ชื่อ + ที่อยู่ + เบอร์โทรตรงกันทุกประการกับ Master Record ที่มีอยู่ (Confidence ≥ 0.95) P5
6 GEO_ANCHOR ACCEPT ที่อยู่ตรง + พิกัด GPS อยู่ในรัศมี 500 เมตร ของ Master Record พิกัดทำหน้าที่เป็น Anchor ยืนยัน P6
7 FUZZY_MATCH REVIEW ชื่อคล้ายกัน (Fuzzy Score ≥ 0.8) แต่ไม่ตรงเป๊ะ ต้องให้ Reviewer ยืนยัน เช่น "สมชาย ใจดี" vs "สมชายใจดี" P7
8 ALL_NEW_WITH_GEO NEW ไม่พบ Candidate ใดเลย แต่มีพิกัด GPS ที่ถูกต้อง สามารถสร้าง Master Record ใหม่ได้ (สถานะ Active) P8

DEFAULT Action: หากไม่ตรงกับกฎใดเลย (ไม่มี Candidate, ไม่มีพิกัด) ระบบจะสร้าง Master Record ใหม่ในสถานะ PENDING และส่งเข้า Q_REVIEW เพื่อให้ Reviewer ตรวจสอบ

7. ระบบ Cache 3 ชั้น

เนื่องจาก Google Apps Script ทำงานในรูปแบบ Serverless (แต่ละ Execution เริ่มต้นจาก State ว่าง) การอ่านข้อมูลจาก Google Sheets ทุกครั้งจึงเป็น Bottleneck หลัก LMDS V5.5 ออกแบบระบบ Cache 3 ชั้น (Three-Layer Cache) เพื่อลดจำนวน Sheet Read Operation จาก ~500 ครั้ง/รัน เหลือ ~50 ครั้ง/รัน

Layer 1: RAM (Global Variables)

ความเร็ว: เร็วที่สุด (~0.1ms) — ข้อมูลอยู่ในหน่วยความจำโปรแกรมโดยตรง

ข้อจำกัด: หายเมื่อ Script Execution จบ ข้อมูลไม่อยู่ระหว่างการรันครั้งต่อไป

การใช้งาน: เก็บ Lookup Table ที่ใช้บ่อย เช่น Person Index, Place Index, Alias Map ที่ต้องอ่านทุกรอบของ Loop

ขนาดสูงสุด: จำกัดโดย Memory Limit ของ Apps Script (~50MB)

Layer 2: CacheService (Script Cache)

ความเร็ว: เร็วปานกลาง (~5ms) — ข้อมูลจัดเก็บใน Google Infrastructure

อายุการใช้งาน: TTL 6 ชั่วโมง — ข้อมูลคงอยู่ข้าม Execution ตราบใดที่ยังไม่หมดอายุ

ขนาดสูงสุด: 100KB per Key — สำหรับข้อมูลที่ใหญ่กว่า 100KB ใช้เทคนิค Chunked Storage (แบ่งเป็นหลาย Key เช่น cache_key_1, cache_key_2, ...) และประกอบกลับเมื่ออ่าน

การใช้งาน: เก็บ Master Data Snapshot, Alias Index, Configuration Cache เพื่อไม่ต้องอ่านจาก Sheet ซ้ำใน Execution ถัดไป

Layer 3: Sheet (Google Sheets)

ความเร็ว: ช้าที่สุด (~100ms per Read) — ต้องผ่าน Google Sheets API

ข้อดี: ถาวร — ข้อมูลคงอยู่ตลอด ไม่หายเมื่อ Script จบหรือ Cache หมดอายุ

การใช้งาน: เป็น Source of Truth สำหรับข้อมูลที่ Cache ใน Layer 1 และ 2 เมื่อ Cache Miss ทั้ง 2 ชั้น ระบบจะอ่านจาก Sheet แล้วเขียนกลับขึ้น Cache ทุกชั้น

กลยุทธ์ Batch Read: อ่านทั้งชีตในครั้งเดียว (getValues()) แทนที่จะอ่านทีละแถว เพื่อลด API Call

Cache Flow: Read Request → Check RAM (L1) → Check CacheService (L2) → Read Sheet (L3) → Write back to L1 + L2 → Return Data

Cache Invalidation: เมื่อมีการเขียนข้อมูลลง Sheet ระบบจะ Invalidate Cache ที่เกี่ยวข้องทั้ง L1 และ L2 ทันที เพื่อป้องกัน Stale Cache

8. กฎการเขียนโค้ด 15+1 ข้อ

LMDS V5.5 กำหนดกฎการเขียนโค้ด 16 ข้อ (15+1) เพื่อให้โค้ดมีคุณภาพ อ่านง่าย และดูแลรักษาได้ กฎเหล่านี้ครอบคลุมตั้งแต่ Naming Convention, Error Handling, Data Integrity ไปจนถึง Performance Optimization

ลำดับ ชื่อกฎ ใจความสำคัญ สถานะ
1 Single Writer ตาราง M_ALIAS ต้องเขียนผ่าน autoEnrichAliasesFromFactBatch_() จุดเดียวเท่านั้น ห้ามเขียนตรงจากฟังก์ชันอื่น PASS
2 Private Suffix ฟังก์ชันภายใน (Private) ต้องลงท้ายด้วย _ (Underscore) เช่น normalizeThaiName_() เพื่อแยกจาก Public API PASS
3 UUID Primary Key ทุก Master Record ต้องใช้ UUID v4 เป็น Primary Key (master_uuid) ห้ามใช้ Auto-increment ID PASS
4 Normalized Storage ข้อมูลต้องถูก Normalize ก่อนเขียนลง Master Table ห้ามเก็บข้อมูลดิบ เช่น ต้องลบคำนำหน้าชื่อก่อนเก็บ PASS
5 Explicit Return ฟังก์ชันทุกฟังก์ชันต้องมี Return Statement ชัดเจน ห้ามใช้ Implicit Undefined Return PASS
6 Error Boundary ทุก Layer ต้องมี try-catch และส่ง Error ไปยัง SYS_LOG ห้ามปล่อยให้ Exception ฟุ้งขึ้นไปโดยไม่จับ PASS
7 Cache-First Read อ่านข้อมูลต้องตรวจ Cache ก่อนเสมอ (L1 → L2 → L3) ห้ามอ่าน Sheet โดยตรงเว้นแต่ Cache Miss PASS
8 Batch Operations อ่าน/เขียน Sheet แบบ Batch เท่านั้น (getValues/setValues) ห้ามอ่านเขียนทีละเซลล์ใน Loop PASS
9 Immutable Audit SYS_LOG เขียนได้อย่างเดียว (Append-only) ห้ามแก้ไขหรือลบ Log ย้อนหลัง PASS
10 Confidence Scoring ทุกการ Match ต้องมี Confidence Score (0.0-1.0) และบันทึกลง FACT_DELIVERY พร้อม Match Rule ที่ใช้ PASS
11 Guard Clause เริ่มฟังก์ชันด้วย Guard Clause (Early Return) สำหรับ Edge Case ห้ามซ้อน if ลึกเกิน 3 ระดับ PASS
12 Const Over Let ใช้ const เป็นค่าเริ่มต้น ใช้ let เฉพาะเมื่อต้อง Reassign ห้ามใช้ var โดยเด็ดขาด PASS
13 Descriptive Naming ชื่อฟังก์ชันและตัวแปรต้องสื่อความหมาย ห้ามใช้ชื่อย่อ เช่น ใช้ findMatchingPerson ไม่ใช่ fmp PARTIAL
14 Function Length ฟังก์ชันหนึ่งฟังก์ชันต้องไม่เกิน 50 บรรทัด หากเกินต้องแบ่งเป็น Sub-function PARTIAL
15 Sheet Schema Lock โครงสร้างคอลัมน์ของ Sheet ต้องตรงกับ Column Mapping ใน SheetUtils ห้ามเพิ่ม/ลดคอลัมน์โดยไม่อัปเดต Mapping PASS
16 No Global Mutable ห้ามใช้ Global Mutable State ยกเว้น Cache L1 (Global Variables) ที่ต้อง Reset ทุก Execution PARTIAL

10. Performance Issues (PERF-001 to PERF-010)

นอกจาก Bug Fix แล้ว LMDS V5.5 ยังได้วิเคราะห์ปัญหา Performance ที่อาจส่งผลกระทบต่อความเร็วในการประมวลผล โดยเฉพาะเมื่อข้อมูลมีขนาดใหญ่ขึ้น ปัญหาเหล่านี้ยังอยู่ในขั้น Plan Only (ยังไม่ได้ดำเนินการแก้ไข) แต่ได้ระบุผลกระทบและแนวทางแก้ไขไว้แล้ว

ID ปัญหา ไฟล์ที่เกี่ยวข้อง ผลกระทบ สถานะ
PERF-001 Sheet Read ทุกครั้งใน Loop แทน Batch Read ทำให้ API Call เกิน 100 ครั้ง/รัน Ingestion, Pipeline รันช้า 3-5x จากที่ควรจะเป็น Plan Only
PERF-002 Fuzzy Match ใช้ Levenshtein Distance ที่ซับซ้อนเกินไปสำหรับการค้นหาเบื้องต้น MatchEngine Match 1 Record ใช้เวลา ~200ms Plan Only
PERF-003 Cache L2 ไม่มี Chunked Read สำหรับข้อมูล > 100KB ทำให้ตกไปอ่าน Sheet ทุกครั้ง CacheManager Cache Miss Rate สูงถึง 40% Plan Only
PERF-004 M_ALIAS Lookup ใช้ Linear Scan แทน Hash Map ทำให้ช้าเมื่อ Alias เกิน 1,000 รายการ M_Alias Lookup Time เพิ่มเป็น O(n) Plan Only
PERF-005 Pipeline รันตามลำดับ (Sequential) ทั้งที่บาง Layer สามารถรันขนาน (Parallel) ได้ Pipeline เวลารันรวมยาวกว่าที่ควรจะเป็น 2x Plan Only
PERF-006 Normalization ทำซ้ำบนข้อมูลเดียวกัน เมื่อ Record ปรากฏในหลาย Batch Normalization, Pipeline เสียเวลา Normalization ซ้ำ ~15% Plan Only
PERF-007 SYS_LOG เขียนทีละแถว แทนที่จะรวบรวมแล้วเขียน Batch เมื่อจบ Pipeline SysLog เพิ่ม Sheet Write Operation ~50 ครั้ง/รัน Plan Only
PERF-008 GeoUtils คำนวณ Haversine ทุกคู่ Candidate แทนที่จะกรองด้วย Bounding Box ก่อน GeoUtils, MatchEngine เพิ่ม Computation 10x จากที่จำเป็น Plan Only
PERF-009 Preflight Audit ตรวจสอบทุกชีตทุกครั้ง แม้จะมีเพียง 2-3 ชีตที่เกี่ยวข้องกับ Pipeline นั้น Preflight เพิ่มเวลา Preflight ~30 วินาที Plan Only
PERF-010 Quality Report สร้างรายงานจาก Sheet Read แบบ Real-time แทนที่จะใช้ Snapshot จาก Cache QualityReport รายงานใช้เวลาสร้าง ~60 วินาที Plan Only

หมายเหตุ: ปัญหา Performance ทั้ง 10 รายการอยู่ในสถานะ Plan Only ยังไม่ได้ดำเนินการแก้ไข แต่ได้วิเคราะห์ผลกระทบและแนวทางแก้ไขไว้แล้ว คาดว่าจะสามารถลดเวลา Pipeline รวมจาก ~8 นาที เหลือ ~3 นาที หากแก้ไขปัญหาทั้งหมด
