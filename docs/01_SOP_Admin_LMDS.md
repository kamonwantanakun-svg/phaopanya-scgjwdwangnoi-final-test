## 1. บทนำ

### 1.1 LMDS คืออะไร?

LMDS (Logistics Master Data System) เป็นระบบจัดการข้อมูลหลักสำหรับการขนส่งสินค้า พัฒนาบน Google Sheets + Google Apps Script ระบบทำหน้าที่หลัก 3 ประการ:

1. **ทำความสะอาดข้อมูล (Data Cleansing)** - ข้อมูลดิบจากคนขับรถมักมีชื่อผิด ที่อยู่ไม่ครบ รูปแบบไม่เป็นมาตรฐาน ระบบจะทำ Normalize อัตโนมัติ
2. **จับคู่ข้อมูล (Data Matching)** - จับคู่ข้อมูลดิบกับ Master Data ที่มีอยู่แล้ว โดยใช้กฎ 8 ข้อ (8-Rule Match Engine)
3. **ค้นหาพิกัดอัตโนมัติ (Coordinate Resolution)** - ค้นหาพิกัด GPS สำหรับจุดส่งมอบสินค้า โดยใช้ระบบแคช 3 ชั้น

### 1.2 บทบาทของ ADMIN

ADMIN มีบทบาทหลักในการ:
- ดึงข้อมูลจัดส่งจากระบบ SCG e-POD ทุกวัน
- ตรวจสอบและอนุมัติข้อมูลที่ระบบจับคู่ไม่มั่นใจ (Q_REVIEW)
- ดูแลข้อมูล Master Data ให้ถูกต้อง
- ตรวจสอบคุณภาพข้อมูลผ่านรายงาน
- บำรุงรักษาระบบเป็นประจำ

### 1.3 ข้อควรระวังสำคัญ

| ข้อควรระวัง | รายละเอียด |
|:---|:---|
| ห้ามแก้ไขข้อมูลใน Sheet โดยตรง | ให้ใช้เมนูระบบเท่านั้น เพื่อป้องกันข้อมูลเสียหาย |
| ห้ามลบแถวใน Master Sheet | ให้เปลี่ยนสถานะเป็น "Inactive" แทน |
| ห้ามเปลี่ยนชื่อ Sheet | ระบบอ้างอิงชื่อ Sheet แบบ Hardcode |
| ตรวจสอบ Q_REVIEW ทุกวัน | ข้อมูลที่ค้างอยู่จะส่งผลต่อความแม่นยำของระบบ |
| สำรองข้อมูลก่อนดำเนินการใหญ่ | ก่อน Run Pipeline หรือ Migration ใหญ่ ให้ Duplicate Spreadsheet ไว้ก่อน |

## 2. การเข้าถึงระบบ

### 2.1 ข้อกำหนดเบื้องต้น

- มีบัญชี Google ที่ได้รับสิทธิ์ Editor บน Spreadsheet ของ LMDS
- ชื่ออีเมลต้องอยู่ในรายชื่อ Admin ที่บันทึกใน ScriptProperties
- เบราว์เซอร์ที่รองรับ: Google Chrome (แนะนำ), Microsoft Edge

### 2.2 การเข้าสู่ระบบ

1. เปิด Google Spreadsheet ของ LMDS จากลิงก์ที่ได้รับ
2. รอให้เมนู **🚚 LMDS V5.5** ปรากฏที่แถบเมนูด้านบน (ใช้เวลา 3-5 วินาที)
3. หากเมนูไม่ปรากฏ ให้รีเฟรชหน้าเว็บ (F5)

### 2.3 การตรวจสอบสิทธิ์

- ระบบจะตรวจสอบสิทธิ์อัตโนมัติเมื่อกดเมนูที่มีผลกระทบต่อข้อมูล
- หากไม่มีสิทธิ์ ระบบจะแสดงข้อความ "คุณไม่มีสิทธิ์ดำเนินการนี้"
- ติดต่อทีม IT เพื่อเพิ่มชื่ออีเมลในรายการ Admin

## 4. SOP งานประจำวัน (Daily Operations)

### SOP-D01: ดึงข้อมูลจัดส่งจาก SCG (รายวัน)

**วัตถุประสงค์:** ดึงข้อมูลงานจัดส่งของวันจากระบบ SCG e-POD เพื่อเตรียมค้นหาพิกัด

**ความถี่:** ทุกวัน (ก่อนเริ่มงาน)

**ขั้นตอน:**

| ลำดับ | การดำเนินการ | หมายเหตุ |
|:---:|:---|:---|
| 1 | เปิด LMDS Spreadsheet | |
| 2 | ไปที่เมนู **🚚 LMDS V5.5 → 🔧 ระบบ & ตั้งค่า → 🔐 ตั้งค่า SCG Cookie** | ใส่ Cookie ที่ดึงจากเบราว์เซอร์ SCG e-POD — ระบบเก็บใน ScriptProperties อย่างปลอดภัย (SEC-001) |
| 3 | ไปที่ Sheet **Input** → วาง ShipmentNos ที่เซลล์ **B3** | คั่นด้วยเครื่องหมายจุลภาค (,) หรือบรรทัดใหม่ |
| 4 | ไปที่เมนู **🚚 LMDS V5.5 → 🟦 กลุ่ม 2 → 📥 ดึงข้อมูล SCG API** | |
| 5 | รอจนกระทั่งข้อความแจ้งเตือน "ดึงข้อมูลสำเร็จ" ปรากฏ | ใช้เวลา 30 วินาที - 3 นาที ขึ้นกับจำนวนข้อมูล |
| 6 | ตรวจสอบ Sheet **ตารางงานประจำวัน** ว่ามีข้อมูล | ต้องเห็น ShipToName, ShipToAddress ฯลฯ |
| 7 | ไปที่เมนู **🚚 LMDS V5.5 → 📍 จับคู่พิกัดวันนี้** หรือ **🟦 กลุ่ม 2 → 📍 จับคู่พิกัด** | |
| 8 | รอจนกระทั่งระบบค้นหาพิกัดเสร็จ | ใช้เวลา 1-5 นาที ระบบมี Auto-Resume หากเกิน 6 นาที |
| 9 | ตรวจสอบคอลัมน์ **LatLong_Actual** ใน Sheet ตารางงานประจำวัน | ดูสีพื้นหลัง: 🟢 เขียว = พบพิกัดคุณภาพสูง, 🟡 เหลือง = Fallback, 🔵 ฟ้า = จาก SCG, 🔴 แดง = ไม่พบ |

**ผลลัพธ์ที่คาดหวัง:**
- Sheet ตารางงานประจำวัน มีข้อมูลงานจัดส่งครบถ้วน
- คอลัมน์ LatLong_Actual มีพิกัด GPS สำหรับรายการที่ระบบจับคู่ได้

## 5. SOP งานจัดการ Master Data

### SOP-M01: โหลดและจับคู่ข้อมูลดิบ (รายสัปดาห์หรือตามความจำเป็น)

**วัตถุประสงค์:** นำข้อมูลดิบเข้าระบบและจับคู่กับ Master Data เพื่อสร้างข้อมูลหลักใหม่หรือเพิ่ม Alias

**ขั้นตอน:**

| ลำดับ | การดำเนินการ | หมายเหตุ |
|:---:|:---|:---|
| 1 | ตรวจสอบว่า Sheet **SCGนครหลวงJWDภูมิภาค** มีข้อมูลดิบ | ข้อมูลจากคนขับรถ |
| 2 | เมนู **🚚 LMDS V5.5 → 🟩 กลุ่ม 1 → Step 1 — โหลดข้อมูลดิบจากแหล่ง** | ระบบจะโหลดข้อมูลที่ยังไม่ได้ประมวลผล |
| 3 | ตรวจสอบจำนวนข้อมูลที่รอประมวลผล | ระบบจะแจ้งจำนวน |
| 4 | เมนู **🚚 LMDS V5.5 → 🟩 กลุ่ม 1 → Step 3 — Match Engine** | ระบบจะประมวลผลทีละแถว (Step 2 Normalize รวมอยู่ใน Pipeline แล้ว) |
| 5 | รอจนกระทั่งประมวลผลเสร็จ | หากข้อมูลมากกว่า 500 แถว ระบบอาจต้อง Resume อัตโนมัติ |

| 7 | ตรวจสอบ Sheet **Q_REVIEW** ว่ามีรายการรอตรวจสอบ | ดูรายละเอียดใน SOP-R01 |

**หมายเหตุ:** หากต้องการทำทั้งโหลดและจับคู่ในครั้งเดียว ให้ใช้ **Run Full Pipeline** แทน

## 6. SOP การตรวจสอบและอนุมัติ Q_REVIEW

### SOP-R01: ตรวจสอบและอนุมัติคิวรีวิว (รายวัน)

**วัตถุประสงค์:** ตรวจสอบข้อมูลที่ระบบจับคู่ด้วยความมั่นใจต่ำ (50-89%) และตัดสินใจแทนระบบ

**ความถี่:** ทุกวัน (ควรทำหลังจาก Run Match Engine)

**หลักการสำคัญ:**
- Q_REVIEW คือกลไก "Human-in-the-Loop" ที่ให้คนตรวจสอบในกรณีที่ระบบไม่มั่นใจ
- การตัดสินใจที่ถูกต้องจะช่วยเพิ่มความแม่นยำของระบบในอนาคต
- ข้อมูลที่ค้างใน Q_REVIEW นานเกินไปจะส่งผลต่อคุณภาพข้อมูล

**ขั้นตอน:**

| ลำดับ | การดำเนินการ | หมายเหตุ |
|:---:|:---|:---|
| 1 | เมนู **🚚 LMDS V5.5 → 🟩 กลุ่ม 1 → 📋 เปิด Review Queue** | ระบบจะพาไปยัง Sheet Q_REVIEW |
| 2 | ดูคอลัมน์ **Status** = "PENDING" | รายการที่รอตรวจสอบ |
| 3 | ดูคอลัมน์ **Priority** - เริ่มจาก HIGH ก่อน | HIGH = คะแนนต่ำสุด, URGENT = มีปัญหาร้ายแรง |
| 4 | วิเคราะห์ข้อมูลในแถว: | |
| | - **raw_person_name** → ชื่อดิบที่ระบบอ่านได้ | |
| | - **raw_place_name** → สถานที่ดิบ | |
| | - **candidate_person** → ข้อมูลที่ระบบเสนอให้จับคู่ | |
| | - **match_score** → คะแนนความมั่นใจ | |
| 5 | เลือก **Decision** จาก Dropdown ในคอลัมน์ Decision: | |
| | - **CREATE_NEW** → สร้างข้อมูลใหม่ใน Master Data | ใช้เมื่อไม่มีข้อมูลที่ตรงกัน |
| | - **MERGE_TO_CANDIDATE** → รวมเข้ากับข้อมูลที่มีอยู่ | ใช้เมื่อเป็นชื่อเดียวกันแต่สะกดต่างกัน |
| | - **ESCALATE** → ส่งต่อให้ผู้บริหารตรวจสอบ | ใช้เมื่อไม่แน่ใจ |
| | - **IGNORE** → ข้ามรายการนี้ | ใช้เมื่อข้อมูลไม่สำคัญ |
| 6 | ระบบจะประมวลผล Decision อัตโนมัติทันที (ผ่าน onEdit) | |
| 7 | ทำซ้ำจนกว่า PENDING ทั้งหมดจะได้รับการตัดสินใจ | |

**เกณฑ์การตัดสินใจ:**

| สถานการณ์ | Decision ที่แนะนำ | เหตุผล |
|:---|:---|:---|
| ชื่อคนเหมือนกัน สะกดต่างกันเล็กน้อย (เช่น สมชาย/สมชัย) | MERGE_TO_CANDIDATE | เป็นคนคนเดียวกัน |
| ชื่อสถานที่เหมือนกัน แต่จังหวัดต่างกัน | CREATE_NEW | เป็นสถานที่คนละแห่ง |
| ข้อมูลไม่ชัดเจน ไม่แน่ใจ | ESCALATE | ให้ผู้บริหารตัดสินใจ |
| ข้อมูลผิดพลาดหรือไม่สมบูรณ์มาก | IGNORE | ไม่คุ้มค่าที่จะประมวลผล |
