## 1. ภาพรวมโปรเจกต์

### LMDS คืออะไร?

**LMDS (Logistics Master Data System)** เป็นระบบจัดการข้อมูลหลัก (Master Data Management) สำหรับงานโลจิสติกส์และการขนส่งสินค้า ที่พัฒนาขึ้นเพื่อใช้งานภายในองค์กร **SCG JWD** (เครือวงการซีเมนต์ไทย ร่วมกับ JWD InfoLogistics) ระบบทำหน้าที่เชื่อมโยงข้อมูลการจัดส่งสินค้าจากระบบ e-POD เข้ากับฐานข้อมูลหลัก และค้นหาพิกัด GPS สำหรับจุดส่งมอบสินค้าโดยอัตโนมัติ

### ข้อมูลทั่วไปของโปรเจกต์

| รายการ | รายละเอียด |
|:---|:---|
| **ชื่อระบบ** | LMDS (Logistics Master Data System) |
| **เวอร์ชัน** | V5.5 (โค้ดจริง = 5.5.014, System Guide = V5.5.014) |
| **แพลตฟอร์ม** | Google Workspace (Google Sheets + Google Apps Script) |
| **ลูกค้า** | SCG JWD Logistics |
| **ขอบเขตการใช้งาน** | การขนส่งสินค้าในประเทศไทย |
| **ภาษาที่ใช้พัฒนา** | JavaScript (Google Apps Script V8) |
| **จำนวนไฟล์โค้ด** | 22 ไฟล์ |
| **จำนวนบรรทัดโค้ด** | ~16,683 บรรทัด |
| **จำนวนฟังก์ชัน** | 312 ฟังก์ชัน |
| **จำนวน Sheet** | 19 แผ่น |
| **สถานะ** | พร้อม Deploy Production (95%) |

## 2. ปัญหาที่แก้ไข

ก่อนมีระบบ LMDS การทำงานด้านข้อมูลการขนส่งสินค้ามีปัญหาสำคัญ 3 ประการ:

### ปัญหาที่ 1: ข้อมูลสกปรก (Dirty Data)

ข้อมูลดิบจากคนขับรถบรรทุกมักมีความไม่สมบูรณ์และไม่ถูกต้อง เช่น:

- **ชื่อผิดสะกด** — "สมชาย" กลายเป็น "สมชัย", "สมชัยย์", "สมใจ"
- **ที่อยู่ไม่ครบ** — ขาดตำบล อำเภอ หรือจังหวัด
- **รูปแบบไม่มาตรฐาน** — บางคนใส่คำนำหน้า "นาย", "คุณ" บางคนไม่ใส่
- **ข้อมูลปนเปื้อน** — เบอร์โทรปนอยู่ในชื่อ, หมายเลขเอกสารปนอยู่ในที่อยู่

**ผลกระทบ:** ข้อมูลที่สกปรกทำให้ไม่สามารถจับคู่กับข้อมูลที่มีอยู่แล้วได้ ส่งผลให้เกิดข้อมูลซ้ำซ้อนและความสับสน

### ปัญหาที่ 2: ไม่มีฐานข้อมูลกลาง (No Single Source of Truth)

ไม่มีตารางข้อมูลหลักที่รวบรวมข้อมูลลูกค้า สถานที่ และจุดส่งมอบ ทำให้:

- ลูกค้าคนเดียวกันปรากฏในระบบด้วยชื่อหลายรูปแบบ
- สถานที่เดียวกันมีหลายรายการในระบบ
- ไม่สามารถติดตามประวัติการจัดส่งของลูกค้าได้อย่างสมบูรณ์

**ผลกระทบ:** การวิเคราะห์ข้อมูลไม่แม่นยำ การตัดสินใจทางธุรกิจขาดข้อมูลที่น่าเชื่อถือ

### ปัญหาที่ 3: ค้นหาพิกัดด้วยมือ (Manual Coordinate Lookup)

เจ้าหน้าที่ต้องค้นหาพิกัด GPS สำหรับจุดส่งมอบสินค้าด้วยตนเอง ทำให้:

- ใช้เวลานาน — ประมาณ 2-5 นาทีต่อ 1 รายการ
- เกิดข้อผิดพลาด — พิกัดไม่แม่นยำหรือผิดสถานที่
- ไม่สามารถรองรับข้อมูลจำนวนมากได้ — 200-500 รายการต่อวัน

**ผลกระทบ:** กระบวนการทำงานช้า ข้อมูลพิกัดไม่แม่นยำ ส่งผลต่อประสิทธิภาพการจัดส่ง

## 3. คุณสมบัติหลักของระบบ

### 3.1 Trinity Framework — โมเดลข้อมูล 3 มิติ

ระบบใช้โมเดล "Trinity" (ตรีเอกา) ในการกำหนดจุดส่งมอบสินค้า โดยต้องประกอบด้วย 3 องค์ประกอบ:

| องค์ประกอบ | ความหมาย | ตัวอย่าง |
|:---|:---|:---|
| **Person** | บุคคล/องค์กร ผู้รับสินค้า | บริษัท สมชาย จำกัด |
| **Place** | สถานที่จัดส่ง | 123 ถ.พหลโยธิน ลาดยาว จ.นนทบุรี |
| **Geo Point** | พิกัด GPS ของสถานที่ | 13.8231, 100.5143 |

จุดส่งมอบ (Destination) ที่สมบูรณ์จะเกิดขึ้นก็ต่อเมื่อมีข้อมูลครบทั้ง 3 องค์ประกอบนี้ เหมือนการระบุที่อยู่ที่ต้องการทั้ง "ใคร" + "ที่ไหน" + "พิกัดอะไร"

### 3.2 ระบบจับคู่อัตโนมัติ (8-Rule Match Engine)

ระบบมีกฎจับคู่ข้อมูล 8 ข้อ ที่ทำงานอัตโนมัติตั้งแต่การจับคู่แบบเต็มรูปแบบไปจนถึงการส่งให้คนตรวจสอบ:

| ระดับ | กฎ | การดำเนินการ |
|:---:|:---|:---|
| CRITICAL | Rule 1: INVALID_LATLNG — พิกัดหาย | REVIEW_INVALID |
| HIGH | Rule 2: LOW_QUALITY — ข้อมูลไม่ครบ | REVIEW |
| HIGH | Rule 3: GEO_PROVINCE_CONFLICT — จังหวัดไม่ตรง | REVIEW |
| MEDIUM | Rule 3.5: NEARBY_PENDING — ระยะพิกัดใกล้เคียง (≤50m AutoMerge, 51-79m Yellow, 80-100m Orange) | ตามระยะ |
| Auto | Rule 4: FULL_MATCH — ตรงทั้งหมด | AUTO_MATCH |
| Auto | Rule 5: GEO_ANCHOR — เจอ Geo+Person เดิม | AUTO_MATCH |
| Auto | Rule 6: FUZZY_MATCH — Score ≥ 90 | AUTO_MATCH |
| New | Rule 7: ALL_NEW_WITH_GEO — ใหม่ทั้งหมด มีพิกัด | CREATE_NEW |
| Review | Rule 8: DEFAULT — ไม่เข้าเงื่อนไข | REVIEW |

ตัวอย่างการทำงาน:
- ข้อมูล "คุณสมชาย" → ระบบ Normalize เป็น "สมชาย" → ค้นหาในฐานข้อมูล → พบ "สมชาย" ที่มีอยู่แล้ว → จับคู่สำเร็จ (คะแนน 95) → **AUTO_MATCH**
- ข้อมูล "บจก.ไทยพาณิชย์ สาขาลาดพร้าว" → Normalize → ค้นหา → พบ "ไทยพาณิชย์ ลาดพร้าว" → จับคู่ (คะแนน 85) → **ส่ง Q_REVIEW**
- ข้อมูล "บริษัท ABC จำกัด" → Normalize → ค้นหา → ไม่พบ แต่มีพิกัด → **CREATE_NEW**

### 3.3 ระบบ Alias (ชื่อแฝง) แบบ Hybrid

ระบบมีตาราง Alias กลาง (M_ALIAS) ที่เก็บชื่อแฝงทุกรูปแบบของลูกค้าและสถานที่:

- **ชื่อสะกดผิด** — "สมชัย" → เก็บเป็น Alias ของ "สมชาย"
- **ชื่อย่อ** — "บจก." → เก็บเป็น Alias ของ "บริษัทจำกัด"
- **คำนำหน้า** — "นายสมชาย" → เก็บเป็น Alias ของ "สมชาย"

เมื่อระบบพบชื่อใหม่ จะเรียนรู้และสร้าง Alias อัตโนมัติผ่าน `autoEnrichAliasesFromFactBatch_()` (Single Writer Pattern) ทำให้การจับคู่ครั้งต่อไปแม่นยำขึ้นเรื่อยๆ นอกจากนี้ยังมีระบบ Fast Track ที่ใช้ M_ALIAS reverse index ค้นหาแบบ O(1) ทำให้การค้นหาพิกัดสำหรับงานประจำวันเร็วขึ้นมาก

### 3.4 ระบบแคช 3 ชั้น (3-Layer Cache)

เพื่อลดการเรียก API ที่มีค่าใช้จ่ายและใช้เวลานาน ระบบใช้แคช 3 ชั้น:

```
ชั้นที่ 1: RAM Cache (เร็วที่สุด, หมดอายุเมื่อ Script จบ)
    ↓ ไม่พบ
ชั้นที่ 2: CacheService (เร็ว, หมดอายุ 6 ชั่วโมง)
    ↓ ไม่พบ
ชั้นที่ 3: Sheet Cache / SYS_TH_GEO (ถาวร, เก็บใน Google Sheets) — *MAPS_CACHE ถูกลบใน V5.5.013 (ใช้ @customFunction formulas แทน); FACT_DELIVERY +2 cols ใน V5.5.014 DRIVER-VERIFIED*
    ↓ ไม่พบ
เรียก API (ช้าที่สุด, มีค่าใช้จ่าย)
```

ผลลัพธ์: การเรียก Google Maps API ลดลงมากกว่า 70% เมื่อเทียบกับการเรียกทุกครั้ง

### 3.5 ระบบ Human-in-the-Loop

ระบบไม่ได้ทำทุกอย่างอัตโนมัติ ในกรณีที่ความมั่นใจต่ำ (คะแนน 50-89) จะส่งข้อมูลเข้า **Q_REVIEW** (คิวรอตรวจสอบ) เพื่อให้เจ้าหน้าที่ตรวจสอบและตัดสินใจ กลไกนี้ช่วย:

- ป้องกันข้อผิดพลาดจากการจับคู่อัตโนมัติที่ผิด
- ให้คนมีส่วนร่วมในกรณีที่สำคัญ
- ระบบเรียนรู้จากการตัดสินใจของคน เพื่อเพิ่มความแม่นยำในอนาคต

### 3.6 ระบบภูมิศาสตร์ไทย (SYS_TH_GEO)

ระบบมีพจนานุกรมภูมิศาสตร์ไทยครอบคลุม **7,537 รายการ** จากทั่วประเทศ ประกอบด้วย:

- ข้อมูลตำบล, อำเภอ, จังหวัด, รหัสไปรษณีย์
- ดัชนีค้นหา (search_key) สำหรับการค้นหาที่รวดเร็ว
- ระบบจำแนกประเภท (note_type) เช่น ตำบลเดียวกันในหลายอำเภอ

ระบบสามารถดึงข้อมูลจังหวัด, อำเภอ, ตำบล จากที่อยู่ภาษาไทยที่เขียนไม่เป็นมาตรฐานได้

## 4. ผลประโยชน์ทางธุรกิจ

### 4.1 ผลประโยชน์เชิงปริมาณ

| ตัวชี้วัด | ก่อนมีระบบ | หลังมีระบบ | การปรับปรุง |
|:---|:---|:---|:---|
| เวลาหาพิกัดต่อรายการ | 2-5 นาที (ด้วยมือ) | < 1 วินาที (อัตโนมัติ) | **>99% ลดลง** |
| ความแม่นยำในการจับคู่ | ไม่มีระบบ | ≥80% อัตโนมัติ | **ใหม่** |
| ข้อมูลซ้ำซ้อน | สูงมาก | ลดลงด้วย Single Source of Truth | **ลดลงมาก** |
| การเรียก Maps API | ทุกรายการ | เฉพาะที่ไม่มีในแคช | **ลด >70%** |
| เวลาประมวลผล Pipeline | N/A (ทำด้วยมือ) | Auto + Auto-Resume | **อัตโนมัติ** |

### 4.2 ผลประโยชน์เชิงคุณภาพ

1. **ข้อมูลเป็นมาตรฐานเดียว (Single Source of Truth)** — ลูกค้าและสถานที่เดียวกันจะถูกรวมเข้าด้วยกัน ไม่ว่าจะถูกเรียกด้วยชื่อใด

2. **ลดงานที่ทำซ้ำ (Reduced Manual Work)** — เจ้าหน้าที่ไม่ต้องค้นหาพิกัด GPS ด้วยตนเองอีกต่อไป ประหยัดเวลาได้หลายชั่วโมงต่อวัน

3. **ข้อมูลดีขึ้นเรื่อยๆ (Self-Improving)** — ระบบเรียนรู้จากข้อมูลใหม่และการตัดสินใจของเจ้าหน้าที่ ทำให้ความแม่นยำเพิ่มขึ้นตามเวลา

4. **ตรวจสอบได้ (Auditable)** — ทุกการจับคู่มีหลักฐาน (match_evidence) และบันทึกประวัติ (SYS_LOG) สามารถตรวจสอบย้อนหลังได้

5. **ต้นทุนต่ำ (Low Cost)** — ใช้ Google Workspace ที่องค์กรมีอยู่แล้ว ไม่ต้องลงทุนโครงสร้างพื้นฐานเพิ่มเติม

### 4.3 ผลตอบแทนจากการลงทุน (ROI)

| รายการ | ต้นทุน | หมายเหตุ |
|:---|:---|:---|
| การพัฒนา | ภายใน (ไม่มีค่าใช้จ่ายเพิ่ม) | ใช้ Google Apps Script |
| โครงสร้างพื้นฐาน | ไม่มี | ใช้ Google Workspace ที่มีอยู่ |
| Google Maps API | มี Fee | แต่ลดลง >70% จากการใช้แคช |
| การบำรุงรักษา | ต่ำ | ระบบทำงานอัตโนมัติเป็นหลัก |
| **ประหยัดเวลา** | **~4 ชั่วโมง/วัน** | ไม่ต้องหาพิกัดด้วยมือ |

## 8. การรับประกันคุณภาพ

### 8.1 กฎการเขียนโค้ด 15+1 ข้อ

โปรเจกต์ปฏิบัติตามกฎการเขียนโค้ด 16 ข้ออย่างเคร่งครัด (16/16 COMPLIANT):

| หมวด | กฎหลัก |
|:---|:---|
| **คุณภาพโค้ด** | Clean Code, Single Responsibility, Full Version Only |
| **ความปลอดภัยของข้อมูล** | No Hardcode Index, Safe Batching, Resumable State |
| **การออกแบบ** | Dependency Map, Zero Hallucination, Namespace Collision |
| **การบำรุงรักษา** | Structured File Naming, Logging, Error Handling |
| **ความปลอดภัย** | Security-First Design |

### 8.2 กระบวนการ Audit 8 รอบ

ระบบผ่านการตรวจสอบคุณภาพ 8 รอบ (CRITICAL → PERF → SECURITY → REVIEW15 → REFACTOR → SYNC → CACHE-FIX → CACHE-CLEANUP):

1. **CRITICAL** — ค้นหา Bug วิกฤตและปัญหาประสิทธิภาพ → พบ 8 Critical → แก้ไขทั้งหมด
2. **PERFORMANCE** — แก้ไขปัญหา Performance 12 รายการ → batch ops, chunked cache
3. **SECURITY** — ตรวจสอบความปลอดภัย → พบ 7 ช่องโหว่ → แก้ไขทั้งหมด
4. **REVIEW15** — ประเมินตามกฎ 16 ข้อ → จาก 8/16 เป็น 13/16
5. **REFACTOR** — ปรับโครงสร้างใหม่ 21 จุด → ลด coupling, เพิ่ม SRP → 16/16 COMPLIANT
6. **Consistency Sync** — แก้ไข 28 doc inconsistencies (doc-only, no code change)
7. **CACHE FIX (P0+P1)** — แก้ไข 9 cache issues (4 P0 data integrity + 5 P1 performance/correctness)
8. **CACHE CLEANUP (P2)** — แก้ไข 6 cache cleanup issues (flush hit_count, flushLogBuffer_ in finally, write-back, orphan cleanup)

## 10. แผนงานในอนาคต

### 10.1 พัฒนาการที่วางแผนไว้

| ลำดับความสำคัญ | ฟีเจอร์ | รายละเอียด | ผลประโยชน์ |
|:---|:---|:---|:---|
| สูง | Dashboard UI | สร้างหน้า Dashboard สำหรับดูสถิติแบบ Real-time | ผู้บริหารเห็นข้อมูลได้ง่ายขึ้น |
| สูง | Auto Cookie Refresh | ต่ออายุ Cookie SCG อัตโนมัติ | ลดงาน Manual |
| กลาง | Email Notification | แจ้งเตือนเมื่อมี Q_REVIEW หรือ Error | ตอบสนองเร็วขึ้น |
| กลาง | REST API Wrapper | สร้าง Web App เพื่อให้ระบบอื่นเรียกใช้ | เชื่อมต่อกับระบบอื่นได้ |
| ต่ำ | Machine Learning | ใช้ ML เพื่อเพิ่มความแม่นยำในการจับคู่ | Auto Match Rate เพิ่มขึ้น |
| ต่ำ | Mobile App | แอปสำหรับคนขับรถอัปเดตพิกัด | ข้อมูลแม่นยำขึ้น |

### 10.2 ข้อเสนอแนะเชิงกลยุทธ์

1. **ขยายการใช้งาน** — ระบบสามารถขยายไปยังสาขาอื่นของ SCG JWD ที่มีปัญหาข้อมูลคล้ายกัน

2. **เชื่อมต่อระบบอื่น** — สามารถพัฒนา API เพื่อให้ระบบ ERP หรือ TMS อื่นดึงข้อมูลพิกัดจาก LMDS ได้

3. **เพิ่ม Data Analytics** — ด้วยข้อมูลที่สะอาดและครบถ้วน สามารถวิเคราะห์พฤติกรรมลูกค้า เส้นทางจัดส่ง และประสิทธิภาพการขนส่งได้

4. **ย้ายไป Database จริง** — หากข้อมูลเพิ่มขึ้นมาก พิจารณาย้ายจาก Google Sheets ไป Cloud SQL หรือ Firestore เพื่อรองรับข้อมูลขนาดใหญ่
